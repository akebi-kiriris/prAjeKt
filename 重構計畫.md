### 存在的問題

| 問題 | 嚴重度 | 說明 |
|------|--------|------|
| **God Component** | 🔴 極嚴重 | `TimelinesView.vue` 2585 行，包含專案管理、任務管理、看板、日曆、AI 生成、留言、檔案、成員分享 |
| **無 Service 層** | 🔴 嚴重 | 每個 View 直接 `import api` 然後自己組裝 API 路徑，無封裝 |
| **只有 1 個 Store** | 🔴 嚴重 | 只有 `auth.js`，其他所有狀態都是 view-local 的 `ref()`，無法跨元件共享 |
| **零元件拆分** | 🟡 中等 | `components/` 只有 Header 和 Sidebar，大量可重用 UI 直接寫在 View 裡 |
| **重複的工具函式** | 🟡 中等 | `formatDate()` 在 4 個 View 各寫了一次 |
| **使用原生 alert/confirm** | 🟡 中等 | 全專案用 `alert()`、`confirm()`，沒有 toast 系統 |
| **無 Composables** | 🟡 中等 | 表單處理、日期格式化等可提取邏輯都寫死在 View 中 |

**Store 範例：**
```javascript
// stores/todos.js
import { defineStore } from 'pinia';
import { ref, computed } from 'vue';
import { todoService } from '../services/todoService';

export const useTodoStore = defineStore('todos', () => {
  const todos = ref([]);
  const loading = ref(false);

  const completedTodos = computed(() => todos.value.filter(t => t.completed));
  const pendingTodos = computed(() => todos.value.filter(t => !t.completed));

  async function fetchTodos() {
    loading.value = true;
    try {
      const res = await todoService.getAll();
      todos.value = res.data.todos;
    } finally {
      loading.value = false;
    }
  }

  async function addTodo(data) {
    const res = await todoService.create(data);
    todos.value.unshift(res.data.todo);
  }

  return { todos, loading, completedTodos, pendingTodos, fetchTodos, addTodo };
});
```

## 切片式開發流程

### 每個功能的標準流程

#### Step 1：資料表設計
- [ ] 分析功能需求，確定所需資料表
- [ ] 設計欄位與資料類型
- [ ] 規劃外鍵關聯與索引

#### Step 2：資料庫遷移
- [ ] 修改或新增 Model（`backend/models/`）
- [ ] 執行 `flask db migrate -m "描述"`
- [ ] 執行 `flask db upgrade`
- [ ] 驗證資料表結構正確

#### Step 3：後端 API 開發
- [ ] 建立或修改 Blueprint（`backend/blueprints/`）
- [ ] 實作 CRUD API
- [ ] 加入驗證與錯誤處理
- [ ] 測試 API（Postman/curl）

#### Step 4：前端開發（重構後的標準流程）
- [ ] 建立 Service（`services/xxxService.js`）— 封裝 API
- [ ] 建立 Store（`stores/xxx.js`）— 管理狀態
- [ ] 建立 Components — 拆分 UI
- [ ] 更新 View — 組裝元件 + 使用 store
- [ ] 建立/更新 Composables — 提取可重用邏輯

#### Step 5：整合測試
- [ ] 前後端聯調測試
- [ ] 邊界情況測試
- [ ] 確認不影響其他功能

### 執行原則

1. 一次只重構一個模組
2. 完成 Service → Store → Component → View 的完整流程
3. 測試通過後再進入下一個
4. 保持其他功能正常運作
